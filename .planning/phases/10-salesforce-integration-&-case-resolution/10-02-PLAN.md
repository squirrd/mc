**Cache location decision:** SQLite database at ~/.mc/cache/case_metadata.db
- WHY: Separate from state database (different access patterns - cache is read-heavy with background writes, state is transaction-heavy)
- WHY: Aligns with v1.0 cache directory structure (~/.mc/cache/)
- WHY: Easier to clear/rebuild cache without affecting container state

**Database schema:**
```sql
CREATE TABLE IF NOT EXISTS case_cache (
    case_number TEXT PRIMARY KEY,
    case_data TEXT NOT NULL,           -- JSON serialized case metadata
    account_data TEXT NOT NULL,        -- JSON serialized account metadata
    cached_at REAL NOT NULL,           -- Unix timestamp
    ttl_seconds INTEGER NOT NULL       -- Cache TTL (default 300 = 5 min)
)
```

**Class structure:**
```python
class CaseMetadataCache:
    """SQLite-based cache with concurrent access support."""

    def __init__(self, cache_dir: Path)
    def _init_db(self) -> None  # Create schema with WAL mode
    def _get_connection(self) -> contextmanager  # Thread-safe connection
    def get(self, case_number: str) -> tuple[dict, dict, bool]  # Returns (case_data, account_data, was_cached)
    def set(self, case_number: str, case_data: dict, account_data: dict, ttl_seconds: int = 300) -> None
    def _is_expired(self, cached_at: float, ttl_seconds: int) -> bool
    def list_all(self) -> list[str]  # List all cached case numbers (for cleanup)
    def delete(self, case_number: str) -> None  # Delete specific cache entry
```

**Implementation requirements:**
- Enable WAL mode: `PRAGMA journal_mode=WAL` in _init_db() for concurrent read/write
- Database path: `{cache_dir}/case_metadata.db` (separate from v1.0 JSON files)
- Connection timeout: 5.0 seconds to prevent indefinite blocking
- Atomic writes: Use `INSERT OR REPLACE` for upserts
- Error handling: Catch sqlite3.OperationalError and retry once, then raise
- JSON serialization: Use `json.dumps(case_data)` to store dicts as TEXT
- Keep v1.0 function `get_case_metadata()` but call new CaseMetadataCache class
- **TTL: Use 300 seconds (5 minutes) as default** (matching REQUIREMENTS.md SF-02)

**WHY these choices:**
- SQLite WAL mode allows concurrent readers while background worker writes (35x faster than file I/O per research)
- Contextmanager pattern ensures connections always close properly (prevents "database locked" errors)
- 5-second timeout prevents blocking on lock contention
- Keep v1.0 function signature for backward compatibility during v2.0 transition
- Separate cache.db from state.db - different access patterns and lifecycle
- 5-minute TTL balances freshness with API rate limit management

**Class structure:**
```python
class CacheManager:
    """Manages case metadata cache with background refresh worker."""

    REFRESH_INTERVAL_SECONDS = 240  # 4 minutes (before 5 min TTL expires)

    def __init__(self, cache: CaseMetadataCache, sf_client: SalesforceAPIClient)
    def start_background_refresh(self, case_numbers: list[str]) -> None
    def stop_background_refresh(self) -> None
    def _refresh_worker(self) -> None  # Background thread target
    def _refresh_case(self, case_number: str) -> None  # Single case refresh

    def get_or_fetch(self, case_number: str) -> tuple[dict, dict, bool]
```

**Implementation requirements:**
- Background worker: `threading.Thread(target=self._refresh_worker, daemon=True)`
- Worker loop: Check all tracked case numbers, refresh if TTL < 60 seconds remaining
- Refresh failure handling: Log error to logger.error(), do NOT raise (worker must continue for other cases)
- Thread synchronization: Use `threading.Lock()` for case_numbers list access
- Graceful shutdown: Set `_stop_event = threading.Event()`, check in worker loop
- get_or_fetch() method: Check cache first, fetch from Salesforce if miss, update cache

**Error handling:**
```python
def _refresh_case(self, case_number: str) -> None:
    try:
        case_data = self.sf_client.query_case(case_number)
        account_data = {"name": case_data["account_name"]}  # Extract from case data
        self.cache.set(case_number, case_data, account_data)
        logger.debug("Refreshed cache for case %s", case_number)
    except SalesforceAPIError as e:
        logger.error("Failed to refresh cache for case %s: %s", case_number, e)
        # Continue - don't stop worker for one failure
    except Exception as e:
        logger.error("Unexpected error refreshing case %s: %s", case_number, e)
```

**WHY these choices:**
- Daemon thread ensures worker stops when main process exits
- 4-minute refresh interval prevents cache from expiring (5 min TTL) during active use
- Lock-based synchronization is simpler than queue-based for small case list
- Log errors but don't raise - one case refresh failure shouldn't break all refreshes
- Per CONTEXT.md: Simple fixed-interval refresh (activity detection deferred to future enhancement)
