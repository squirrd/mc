## Decisions Made

**Config key naming:**
- Changed `offline_token` → `rh_api_offline_token` to clearly indicate it's for Red Hat API authentication
- Previous name was ambiguous and didn't indicate purpose

**Backwards compatibility strategy:**
- Validation accepts both `rh_api_offline_token` (new) and `offline_token` (old)
- Code reads new key first via `.get()`, falls back to old key if not present
- Deprecation warning logged when old key detected (helps users know to migrate)
- No breaking changes for existing users

**Migration approach:**
- New installations get rh_api_offline_token automatically (wizard updated)
- Existing users can keep old key (backward compat) or rename at their convenience
- Documentation shows new key with migration note in INSTALL.md

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed socket_path TOML serialization error**
- **Found during:** Task 1 (running config tests)
- **Issue:** Default config had `socket_path: None` which is not TOML-serializable, causing `TypeError: Object of type 'NoneType' is not TOML serializable` when saving config
- **Fix:** Changed `socket_path: None` → `socket_path: ""` (empty string for auto-detection)
- **Files modified:** src/mc/config/models.py (line 25), validation logic (lines 73-76)
- **Verification:** Config tests pass, TOML serialization works, PodmanClient handles empty string correctly (same behavior as None)
- **Committed in:** abd7085 (Task 1 commit)

**Analysis:** This was a pre-existing bug that wasn't caught because the test suite wasn't exercising the TOML serialization path with default config. The fix is safe because PodmanClient already handles both None and empty string as "auto-detect socket path" (lines 115-118 in podman.py use truthiness check).
