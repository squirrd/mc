<objective>
Implement intelligent patch version bumping based on digest comparison. When building a new image, the script queries the registry for the latest patch version matching the minor version from versions.yaml, builds the image, compares digests, and automatically increments the patch version and pushes to the registry only when content has changed.

Purpose: Eliminate manual version management errors while preventing unnecessary registry bloat from unchanged images. Registry becomes the source of truth for patch versions, preventing conflicts when multiple developers build.

Output: Fully automated versioning system that bumps patch versions on content changes, validates semantic versioning compliance, detects version conflicts, and auto-pushes new versions to quay.io.
</objective>

<task type="auto">
  <name>Task 2: Implement auto-versioning logic with digest-based bumping</name>

  <files>container/build-container.sh</files>

  <action>
Enhance build-container.sh with auto-versioning logic following the workflow from CONTEXT.md and RESEARCH.md patterns:

**1. Add semantic version validation function:**
- Create validate_semver() function using regex: ^(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)$
- Rejects leading zeros (01.2.3), missing components (1.2), non-integers (1.a.3)
- Use for validating calculated versions before tagging

**2. Add patch version finder function:**
- Create find_latest_patch() function that:
  - Takes minor_version parameter (e.g., "1.0")
  - Calls existing query_latest_registry_version() but filters to minor version prefix
  - Uses: skopeo list-tags | jq | grep "^${minor_version}\.[0-9]+$" | sort -V | tail -1
  - Returns latest patch number (17 from "1.0.17") or -1 if no tags found
- Reuse existing exponential backoff retry logic from Phase 23

**3. Add auto-versioning orchestration function:**
- Create auto_version_and_push() function that implements CONTEXT.md workflow:

  a) Extract minor version from versions.yaml (modify extract_and_validate_versions):
     - Change IMAGE_VERSION extraction to read x.y format (not x.y.z)
     - Validate with regex: ^[0-9]+\.[0-9]+$
     - Parse into MAJOR, MINOR variables using IFS='.' read

  b) Query registry for latest patch:
     - Call find_latest_patch("${MAJOR}.${MINOR}")
     - Store CURRENT_PATCH (e.g., 17, or -1 for first build)

  c) Build image locally (reuse existing perform_build logic):
     - Build mc-rhel10:temp tag (not :latest yet)
     - Extract local digest: podman inspect mc-rhel10:temp | jq -r '.[0].Digest'

  d) Compare digests:
     - If CURRENT_PATCH >= 0: Get registry digest for "${MAJOR}.${MINOR}.${CURRENT_PATCH}"
     - Use existing get_registry_digest() function from Phase 23
     - Compare local vs registry digest
     - If match: Log "Image unchanged, skipping bump and push" → return 0 (no-op)

  e) Calculate new version (digest differs or first build):
     - NEW_PATCH=$((CURRENT_PATCH + 1))  # Handles -1 → 0 for first build
     - NEW_VERSION="${MAJOR}.${MINOR}.${NEW_PATCH}"
     - Validate with validate_semver()

  f) Detect version conflicts:
     - Use existing check_version_exists() to verify NEW_VERSION not on registry
     - If exists: Fail with error (race condition or logic bug)

  g) Tag and auto-push:
     - Tag mc-rhel10:temp as both mc-rhel10:${NEW_VERSION} and mc-rhel10:latest
     - Run: podman push mc-rhel10:${NEW_VERSION} docker://${REGISTRY_REPO}:${NEW_VERSION}
     - Run: podman push mc-rhel10:latest docker://${REGISTRY_REPO}:latest
     - Log "Published ${NEW_VERSION} to ${REGISTRY_REPO}"

**4. Update main execution flow:**
- Replace perform_build() with auto_version_and_push() after registry query
- Keep dry-run mode: Skip registry query, show would-be version calculation
- Update JSON output to include: new_version, bumped (true/false), pushed (true/false)
- Preserve existing --verbose, --json, --registry flags

**5. Error handling:**
- Fail fast on registry query failure (existing behavior from Phase 23)
- Fail fast on version conflict (version already exists)
- Fail fast on invalid semver format
- Fail fast on podman push errors (auth, network, quota)

**What NOT to do:**
- Don't add --push flag (CONTEXT.md specifies auto-push on digest differ)
- Don't add --force-bump flag (digest comparison is always authoritative)
- Don't store patch version in versions.yaml (registry is source of truth)
- Don't use file-based change detection (digest comparison only)

Use RESEARCH.md Pattern 5 (Digest-Based Bump Decision) and Complete Auto-Versioning Function as reference implementations. Reuse Phase 23's skopeo retry logic and credential handling patterns.
  </action>
