Create terminal attachment orchestration module integrating container lifecycle, shell customization, and terminal launching.

**src/mc/terminal/attach.py:**

should_launch_terminal() -> bool:
- Check if stdout is a TTY using sys.stdout.isatty()
- Return True if interactive terminal, False if piped/redirected
- Per RESEARCH.md Pattern 3 (TTY detection)
- Use stdlib sys module (Python implementation - see STACK.md)

attach_terminal(case_number: str, config_manager, salesforce_client, container_manager):
- Complete workflow for terminal attachment to case container
- Steps:
  1. Validate case number format (8 digits, CONTEXT.md requirement)
  2. Check TTY - if not interactive, raise error with message "mc case command requires interactive terminal"
  3. Fetch case metadata from Salesforce (via salesforce_client from Phase 10)
  4. Check if container exists (via container_manager.state)
  5. If container missing: auto-create using container_manager.create() with brief message "Creating container..."
  6. If container stopped: auto-start using container_manager (existing functionality from Phase 11)
  7. Get workspace path and customer name from case metadata
  8. Generate custom bashrc using write_bashrc() from shell module
  9. Build podman exec command: `podman exec -it --env BASH_ENV={bashrc_path} --env PS1=[MC-{case_number}] \\w\\$ {container_id} /bin/bash`
  10. Build terminal window title: "{case_number} - {customer_name} - {description}" (CONTEXT.md format)
  11. Get terminal launcher using get_launcher() from launcher module
  12. Launch terminal with command and title (non-blocking via launcher.launch())
  13. Return immediately (host terminal returns to prompt)
- Error handling:
  - Salesforce API failure: "Failed to fetch case metadata. Check network and credentials."
  - Container creation failure: Show Podman error + MC interpretation (CONTEXT.md requirement)
  - Terminal launch failure: "Terminal launch failed. Run: mc --check-terminal" (brief with fix hint per CONTEXT.md)
- Logging: Log all errors to MC log file (use logging module from mc.utils.logging)

build_exec_command(container_id: str, bashrc_path: str, case_number: str) -> str:
- Helper to build podman exec command string
- Return: `podman exec -it --env BASH_ENV={bashrc_path} --env PS1=[MC-{case_number}] \\w\\$ {container_id} /bin/bash`
- Follow RESEARCH.md Pattern 2 (container attachment with custom environment)

build_window_title(case_number: str, customer_name: str, description: str) -> str:
- Helper to format window title per CONTEXT.md
- Truncate description if too long (keep under 100 chars total)
- Return: "{case_number} - {customer_name} - {description}"

Use established MC patterns: type hints, docstrings, error logging, integration with existing modules.
Import ContainerManager from mc.container.manager, CaseResolver from mc.integrations.salesforce.

**IMPORTANT - Python implementation:**
- Use sys.stdout.isatty() for TTY detection (Python stdlib)
- Use subprocess module for podman exec (Python stdlib)
- RESEARCH.md shows Go patterns for reference, but implement using Python equivalents
