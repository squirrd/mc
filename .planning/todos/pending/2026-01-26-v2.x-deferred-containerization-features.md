# Deferred Containerization Features (v2.x)

**Created:** 2026-01-26
**Context:** During v2.0 milestone planning, we scoped down to MVP containerization. This captures the deferred features from the original high-level design.

## Auto-Update System (v2.1 candidate)

**3-plane versioning model:**
- Host mc CLI auto-updates silently from PyPI/GitHub
- Container image auto-pulls from quay.io on every invocation
- Container mc CLI baked into image (immutable per image)

**Commands:**
```bash
mc update                    # update host mc to latest
mc update <version>          # update to specific version
mc update list               # list recent available versions
```

**Image tag resolution:**
- Check config for `MC_IMAGE_TAG` (defaults to `latest`)
- Pull if newer digest available
- Allow pinning to specific versions for stability

**mc-update script:**
- Separate simple Python script for self-update
- Minimal dependencies so it rarely fails
- Fallback when main mc is broken

## Mount Policy Engine (v2.2 candidate)

**Declarative mount configuration:** `~/.mc/mounts.yaml`

```yaml
version: 1
defaults:
  users:
    admin: 1000
    user: 2000

mounts:
  - name: system
    host: /mc/system
    container: /system
    perms:
      admin: ro
      user: no

  - name: shared
    host: /mc/shared
    container: /shared
    perms:
      admin: rw
      user: ro

  - name: tools
    host: ~/.config/mc/tools
    container: /tools
    perms:
      admin: rw
      user: ro

  - name: case-root
    host: /mc/cases/{customer}/{case}
    container: /case
    perms:
      admin: rw
      user: rw
```

**Features:**
- Dynamic variable substitution: `{customer}`, `{case}` resolved from metadata
- Multi-user permissions (admin/user roles inside container)
- Permission enforcement via `podman --userns=keep-id` and POSIX/ACLs
- Mount types: RO, RW, or hidden (`no`)

## Automatic Housekeeping (v2.3 candidate)

**Auto-stop:**
- Stop containers after 48 hours inactivity (configurable)
- Stop when case closed in Salesforce
- Stop when Jira closed
- Stop when no case/Jira activity for N days (configurable)

**Auto-cleanup:**
- Remove stopped containers after 5 days (configurable)
- Compact case data after 6 weeks if container doesn't exist (configurable)

**Compaction:**
- `mc case compact <caseNumber>` removes duplicate data
- Rule-driven via `~/.mc/compact.yaml`
- Preserves user-generated notes
- Removes data that exists in authoritative sources

## Customer/Case Profiles (TBD - may not be needed)

**Customer profile (`customer.yaml`):**
- Customer-specific invariants: clusters, Jira projects, default image tag, mount overrides
- Shared across all cases

**Case profile (`case.yaml`):**
- Case-specific overrides: cluster assignment, Jira tickets, image tag, mount overrides
- Generated per case; inherited defaults from customer profile

**Note:** During planning, we questioned if profiles are needed. May use alternative approach (environment variables, database, Salesforce as source of truth).

## Implementation Notes

**Versioning:** These features should be separate milestones, not bundled. Each adds significant complexity.

**Priority order (suggested):**
1. Extended tools (incremental value, low risk)
2. Auto-update system (high value for image freshness)
3. Automatic housekeeping (nice QoL, low risk)
4. Mount policy engine (complex, only if multi-user needed)
5. Profiles (only if simpler approaches insufficient)

**Research needs:**
- Auto-update: How do other CLIs handle self-update? (brew, rustup, etc.)
- Housekeeping: Common patterns for container lifecycle automation
- Mount policy: Rootless podman permission models and userns complexity
