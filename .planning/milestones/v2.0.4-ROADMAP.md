# Milestone v2.0.4: Foundation

**Status:** ✅ SHIPPED 2026-02-19
**Phases:** 26-28
**Total Plans:** 6

## Overview

Build version checking infrastructure and configuration foundation for automatic updates (MC CLI only in this milestone). This foundation enables the v2.0.5 MC Auto-Update milestone to implement actual update installation logic.

## Phases

### Phase 26: Configuration Foundation

**Goal**: Extend TOML config system with version management fields and safe concurrent write patterns
**Depends on**: Phase 25 (v2.0.3 shipped)
**Plans**: 2 plans

Plans:

- [x] 26-01: Extend config models with [version] section and atomic write implementation
- [x] 26-02: Test suite for version config functionality and backward compatibility

**Details:**

Extended TOML config system with [version] section for pinned_mc and last_check fields. Implemented atomic write method using tempfile.NamedTemporaryFile + os.replace() pattern for thread safety. Created version config helpers (get_version_config, update_version_config) with backward-compatible defaults. All tests passing with full coverage of defaults, updates, partial updates, and TOML structure validation.

**Success Criteria Achieved:**
1. ✓ TOML config persists pinned_mc_version field and last_version_check timestamp
2. ✓ Config writes are atomic (temp file + rename pattern prevents partial writes and corruption)
3. ✓ Config read operations return defaults when version management fields are missing (backward compatibility)

**Requirements Satisfied:** UCTL-05, UCTL-06, UCTL-10

### Phase 27: Runtime Mode Detection

**Goal**: Detect container vs host execution context to prevent auto-update in containerized environments
**Depends on**: Phase 26 (config foundation exists)
**Plans**: 2 plans

Plans:

- [x] 27-01: Add fallback container detection and auto-update guard with Rich messaging
- [x] 27-02: Test suite for fallback detection and auto-update guard behavior

**Details:**

Layered container detection using MC_RUNTIME_MODE environment variable (primary) with filesystem indicator fallback (/run/.containerenv, /.containerenv, /.dockerenv). Auto-update guard (should_check_for_updates) blocks version checks in agent mode with informational Rich banner message. Avoided fragile cgroups parsing per research findings. All tests passing with full coverage of detection scenarios and auto-update blocking.

**Success Criteria Achieved:**
1. ✓ System correctly identifies when running in container (agent mode) vs host
2. ✓ Auto-update functionality is disabled when running in container mode
3. ✓ Container mode shows informational message: "Updates managed via container builds"
4. ✓ Runtime mode detection works across different container runtimes (podman, docker)

**Requirements Satisfied:** RTMD-01, RTMD-02, RTMD-03

### Phase 28: Version Check Infrastructure

**Goal**: Non-blocking GitHub API integration with throttling and caching for MC CLI version checking
**Depends on**: Phase 27 (runtime mode detection prevents container updates)
**Plans**: 2 plans

Plans:

- [x] 28-01: VersionChecker core with daemon threads, GitHub API integration, and PEP 440 comparison
- [x] 28-02: CLI integration, manual override command, and comprehensive test suite

**Details:**

VersionChecker class (312 lines) with daemon thread lifecycle and atexit cleanup. GitHub API integration with ETag conditional requests (If-None-Match header) and exponential backoff retry logic. PEP 440 version comparison using packaging library. Hourly check throttle (3600s) with 24-hour extension on rate limits. Once-per-day notification display to stderr. CLI integration at startup (non-blocking background check) plus manual override command (mc version --update). 29 tests passing with 86% coverage.

**Success Criteria Achieved:**
1. ✓ System checks GitHub releases API for latest MC CLI version without blocking commands
2. ✓ Version checks respect hourly throttle (no check if less than 1 hour since last check)
3. ✓ System uses ETag conditional requests and caches version data with timestamps
4. ✓ Version comparison uses PEP 440-compliant logic (packaging library)
5. ✓ Network failures are handled gracefully (show warning, continue with current version)

**Requirements Satisfied:** VCHK-01, VCHK-03, VCHK-04, VCHK-05, VCHK-06, VCHK-07, VCHK-08

## Milestone Summary

**Key Decisions:**
- Use daemon threads with atexit cleanup (not asyncio) - matches existing CacheManager pattern
- Store ETag, latest_known, latest_known_at, last_status_code in [version] section for conditional requests
- Retry with tenacity for transient failures, fail fast on 4xx errors
- Separate hourly check throttle (3600s) from daily notification throttle (86400s)
- Extend check throttle to 24 hours when rate limited (last_status_code == 403)
- Use MC_RUNTIME_MODE environment variable as primary container detection (explicit contract)
- Fallback to filesystem indicators for edge cases (defensive /run/.containerenv, /.dockerenv)
- Block auto-update in agent mode with informational Rich message
- Avoid cgroups parsing (fragile, deprecated in cgroups v2, broken by Linux 6.12+)
- Use temp file in same directory for atomic replace (same filesystem guarantee)
- Unix epoch float timestamps for consistency with cache.py and auth.py patterns

**Issues Resolved:**
- Configuration foundation for version management established
- Runtime mode detection prevents containerized auto-updates
- Non-blocking version checking infrastructure complete

**Tech Debt Incurred:**
- UCTL-09 (file locking) explicitly descoped - single-process assumption per CONTEXT.md
- should_check_for_updates() exported but unused - CLI uses direct get_runtime_mode() check instead
- update_version_config() orphaned - VersionChecker needs 5-field updates, uses manual dict construction
- check_for_updates() convenience function exported but unused

**Milestone Audit:** PASSED (13/14 requirements - 1 descoped by design)
